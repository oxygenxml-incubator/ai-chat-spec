Abstract

This proposal defines an XPath 3.1 extension function for invoking AI language models. It defines a generic chat-completion function (ai:chat) that sends a conversation to an AI backend and returns the response together with the full conversation history, so multi-turn workflows are possible. The module is designed to be compatible with XSLT 3.0, XQuery 3.1, Schematron, SQF, XProc 3.0, and any other XPath 3.1 host language, and to be independent of any specific AI provider.

1 Introduction

AI language models are increasingly useful inside XML processing pipelines, for generating content, checking meaning, summarising documents, and making decisions that require natural-language understanding. This specification defines a minimal, portable XPath 3.1 API that exposes chat-completion capabilities without tying author code to any specific AI provider or tool.

The design follows the same philosophy as other EXPath modules: define a stable namespace, a small set of well-typed functions, and leave implementation details (connector configuration, model selection, authentication) to the host environment. An XSLT stylesheet or Schematron schema that calls ai:chat should run unchanged on any conforming XPath processor, regardless of the AI backend configured behind it.

1.1 Namespace conventions

The module defined by this document defines functions in the namespace http://expath.org/ns/ai. In this document, the ai prefix, when used, is bound to this namespace URI.

Error codes are defined in the same namespace as the functions (http://expath.org/ns/ai), and use the same ai prefix. They are therefore not in the http://www.w3.org/2005/xqt-errors namespace, which is reserved for specification-defined errors and for the err:* error-information variables available inside a catch handler.

Example namespace declaration for XSLT:

xmlns:ai="http://expath.org/ns/ai"

1.2 Error management

Error conditions are identified by a QName in the http://expath.org/ns/ai namespace. When an error condition is reached and on-error is 'throw' (the default), a dynamic error is raised with the corresponding error code. When on-error is 'return', the same condition is instead reported in the result map through the error and error-code keys (see ).

The error codes are raised in the same namespace as the ai:chat function that was called. An implementation that also exposes ai:chat under an additional, implementation-defined namespace raises the errors in that same namespace, so the same prefix can be used to call the function and to catch its errors.

Defined error codes:

  • ai:backend-error — The AI backend returned an error, was unreachable or timed out, or could not produce a completion. This includes the case where the requested model is unavailable; an implementation MAY raise a more specific error for that case.

  • ai:invalid-argument — The $messages sequence is empty or contains an item that cannot be interpreted as a message, or a value in $opts has a type the implementation cannot use. An implementation MAY coerce a reasonable value (for example, the string value of a node or atomic item) rather than raising this error; it MUST raise ai:invalid-argument only when no usable interpretation exists.

  • ai:not-available — AI functionality is disabled or not configured in the host environment.

  • ai:access-denied — Access to a resource required for the call is blocked by host policy (for example, a document excluded by an ignore list).

  • ai:error — A general fallback for any failure not covered by a more specific code above. An implementation MUST report every failure with one of the codes in this list, using ai:error when no more specific code applies, so that callers can always rely on a code being present (and, in 'return' mode, on receiving a result map).

1.3 Design goals

  • Provider independence. The functions do not expose provider-specific parameters in the type signature. Provider selection and authentication are environment configuration, not author code.

  • Composability. The ?messages array returned by ai:chat can be passed directly as part of the next call's $messages sequence, so multi-turn conversations need no external state.

  • Minimal surface area. A single function covers all common AI integration patterns. Authors who only need a single-turn string result can call ai:chat('prompt')?response, while authors building multi-turn pipelines use the full map.

2 Message Format

Both functions accept a sequence of message items. A message item is one of the following:

  • xs:string — treated as a user message. The string value becomes the content of the message.

  • map(*) with at least a role key (xs:string) and a content key (xs:string). Known role values are system, user, assistant, and tool. The handling of any other role value is implementation-defined; an implementation MAY map an unrecognised role to user.

  • array(*) whose members are message maps as described above — the array is auto-expanded into individual message items. This allows a ?messages array from a prior ai:chat call to be spliced directly into the next call's sequence without further processing.

A tool-role message map additionally carries a tool_call_id key (xs:string) identifying the preceding tool-call turn it answers. An assistant-role map that initiated tool calls carries a tool_calls key containing the serialised JSON from the model response. Both extra keys are preserved transparently when the ?messages array is passed to a subsequent ai:chat call.

3 The ai:chat Function

Sends a conversation to the AI backend and returns a map containing the assistant reply and the full updated conversation history.

3.1 Signature

ai:chat($messages as item()*, $opts as map(*)?) as map(*)

3.2 Parameters

3.2.1 $messages

One or more message items (see 2 Message Format) forming the conversation to send. The sequence must contain at least one item. The items are assembled into an ordered message list before the call is dispatched: arrays are expanded in-place, strings become user maps. The ordering and composition of messages are forwarded to the backend as-is; sequence constraints (such as which role may appear last) are backend-specific and reported via ai:backend-error.

3.2.2 $opts

An optional map of call-level settings. All keys are optional. Unknown keys are silently ignored or reported in ?warnings at implementation discretion. Recognised keys:

  • model (xs:string) — The model or engine identifier to use for this call, overriding the environment default.

  • temperature (xs:double) — Sampling temperature in the range 0–2. Higher values produce more varied output.

  • max-tokens (xs:integer) — Maximum number of tokens in the completion.

  • max-completion-tokens (xs:integer) — Maximum tokens including reasoning tokens, for models that perform explicit reasoning before responding.

  • reasoning-effort (xs:string) — Hints the amount of reasoning the model should perform. Recognised values: 'low', 'medium', 'high'.

  • stop (xs:string* or array(xs:string)) — Up to four stop sequences. The model stops generating when any stop sequence appears in the output.

  • cache (xs:boolean) — When true() (the default), the implementation may return a cached result for an identical prior call, avoiding redundant API round-trips.

  • on-error (xs:string) — Controls error handling. 'throw' (default) raises a dynamic error in the http://expath.org/ns/ai namespace on failure. 'return' suppresses the dynamic error; the returned map instead carries error and error-code keys describing the failure.

  • tools (xs:string?) — Full tool definitions available to the model for this call, serialised as a JSON string.

3.3 Return Value

The function returns a map(*) with the following keys:

  • response (xs:string?) — The final assistant reply as a plain string. Absent when an error occurred and on-error is 'return'.

  • messages (array(map(*))) — The complete conversation history after this call: all input messages plus the assistant reply appended. Each member is a message map with at minimum role and content keys. This array may be passed directly as part of the $messages sequence of a subsequent ai:chat call to continue the conversation.

  • error (xs:string?) — Present only when on-error is 'return' and the call failed. Contains a human-readable description of the failure.

  • error-code (xs:string?) — Present only when on-error is 'return' and the call failed. Contains the local name of the error code that would otherwise have been raised (for example, 'backend-error').

  • warnings (array(xs:string)) — Zero or more non-fatal diagnostic messages produced during the call. Absent when empty.

3.4 Tool and Function Calling

The $opts?tools key enables the model to invoke caller-defined functions during the conversation. Its value is a JSON string containing an array of tool definitions. Each definition follows the structure:

[{
  "type": "function",
  "function": {
    "name": "get_document_content",
    "description": "Read content from a file or URL.",
    "parameters": {
      "$schema": "https://json-schema.org/draft/2020-12/schema",
      "type": "object",
      "properties": {
        "docURL":     { "type": "string" },
        "start_line": { "type": "integer" },
        "end_line":   { "type": "integer" }
      },
      "required": ["docURL", "start_line", "end_line"]
    }
  }
}]

The parameters object is a JSON Schema (2020-12) object describing the arguments the model must supply when invoking the tool.

The tools value MUST be a list of full tool definitions as shown above (each an object with a function name). A tools value that is not a valid list of full tool definitions is ignored and reported in warnings rather than raising an error.

When tools are provided, the implementation MUST execute the following loop transparently on behalf of the caller:

  1. Submit the conversation (including tool definitions) to the model.

  2. If the model returns a tool-call request, execute the named tool with the supplied arguments.

  3. Append the tool result as a tool-role message and re-submit the conversation.

  4. Repeat until the model returns a final text response.

From the XPath author's perspective, this loop is a single ai:chat call returning a single map. The ?response key holds the final text response. All intermediate tool-call and tool-result turns are preserved in the ?messages array, making the full reasoning trace available for logging or inspection.

Tool calling bridges both integration directions: an ai:chat call equipped with XML-tool definitions (XSLT, Schematron, XProc) represents AI orchestrating XML tools from within an XML pipeline, with both directions active in a single function call.

3.5 Examples

Single-turn, plain-string result:

ai:chat('What is a DITA shortdesc?')?response

With explicit system prompt:

ai:chat((
  map{'role': 'system', 'content': 'You are a DITA expert.'},
  'What is a shortdesc?'
))?response

Multi-turn (the ?messages array is auto-expanded):

let $r1 := ai:chat('What is a shortdesc?')
let $r2 := ai:chat(($r1?messages, 'Give me an example.'))
return $r2?response

With options (specific model and non-throwing error handling):

let $r := ai:chat(
  (map{'role':'system','content':'You are a DITA expert.'}, 'What is a shortdesc?'),
  map{'model':'gpt-4o', 'on-error':'return'}
)
return if (map:contains($r, 'error'))
       then 'Error: ' || $r?error
       else $r?response

Use inside an XSLT template to generate a short description:

<xsl:value-of select="
  ai:chat('Generate a DITA shortdesc element for: ' || .)?response"/>

Use inside a Schematron rule to perform a semantic check:

<sch:assert test="
  ai:chat('Does this short description accurately summarise the following topic? '
          || .. || ' Shortdesc: ' || .
          || ' Answer yes or no.')?response = 'yes'">
  Short description may not match the topic.
</sch:assert>

4 Conformance

A conforming implementation of this module:

  • MUST bind the prefix ai to the namespace URI http://expath.org/ns/ai and make the function ai:chat available under that namespace.

  • MUST accept all message item types described in 2 Message Format. It MAY coerce other items to their string value, and MUST raise ai:invalid-argument when $messages is empty or an item cannot be used as a message.

  • MUST return a map with at minimum the response and messages keys on a successful call.

  • MUST raise ai:backend-error on an AI backend failure when on-error is 'throw', and when on-error is 'return' MUST instead return the error and error-code keys.

  • MUST normalise $messages items before dispatching: expand array(*) items into their individual member maps, and convert xs:string items into user-role message maps.

  • MAY support additional $opts keys beyond those specified here, provided they do not conflict with the defined keys.

  • MAY connect to any AI backend; the choice of backend and its configuration is outside the scope of this specification.
